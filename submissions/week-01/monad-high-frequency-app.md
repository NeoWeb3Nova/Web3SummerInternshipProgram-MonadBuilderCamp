# Monad 理解｜为什么 Monad 体验不同

## 任务题目

选择一个适合 Monad 的高频交互场景，并说明为什么它可能更适合在 Monad 上构建。

本次选择的场景：**Onchain Quest Arena：实时任务 / 排行榜 / 小游戏积分系统**

一句话描述：
> 用户每天完成链上小任务、小游戏挑战、社交互动或 AI Agent 任务后，系统实时发放积分、徽章、排名变化和赛季奖励。它像“Web3 版 Duolingo + Zealy + 小游戏排行榜”，但关键行为和奖励结算可以上链。

---

## 1. Research：为什么 Monad 更适合这个高频交互应用（约 300 字）

我认为 Monad 适合构建“Onchain Quest Arena”这类实时任务和排行榜产品。这个场景天然需要高频交互：用户可能每天签到、完成任务、参与小游戏、提交挑战结果、领取积分、查看排名变化，还可能和 AI Agent 或其他玩家频繁互动。如果每一次积分更新、徽章领取、排名刷新都要等待很久，用户会感觉产品像“卡住了”；如果手续费很高，原本轻量的任务行为就会变成心理负担，用户不会愿意为了一个小徽章或一次小游戏支付明显成本。

Monad 的价值不只是“更快”，而是它可能让链上交互接近普通互联网产品的反馈节奏。0.4 秒级出块、高吞吐和并行执行能力，适合承载大量小额、重复、实时反馈型状态更新；EVM 兼容性则让开发者可以继续使用 MetaMask、Foundry、Hardhat、Viem、Wagmi 等熟悉工具，降低从以太坊生态迁移和快速做 demo 的成本。

不过，这个产品不应该把所有数据都上链。普通浏览记录、UI 状态、临时草稿、推荐算法等用数据库即可。真正适合上链的是：用户身份、关键任务完成证明、积分结算、徽章/NFT、赛季奖励、排行榜快照和反作弊可验证记录。这样既保留 Web2 产品的流畅体验，又用链上记录提供公开、可验证、可组合的 Proof of Participation。

## 2. Tech：高频交互 Demo 功能清单

Demo 名称：**Monad Quest Arena**

### A. 用户与钱包

- [ ] 连接钱包（MetaMask / WalletConnect）
- [ ] 检测 Monad Testnet 网络
- [ ] 一键提示用户切换到 Monad Testnet
- [ ] 显示用户地址、昵称、头像、当前积分

### B. 任务系统

- [ ] 每日签到任务：每天可 claim 一次积分
- [ ] 链上交互任务：完成一次指定合约调用后获得积分
- [ ] 社交任务：提交 X / Discord / Farcaster 任务链接，由后端或人工审核后发放积分
- [ ] 学习任务：完成一篇 Monad 概念问答后领取徽章
- [ ] 限时任务：例如 10 分钟内完成小游戏挑战

### C. 高频交互小游戏

- [ ] 轻量小游戏：例如“Gas Runner”“区块接力”“交易排序挑战”
- [ ] 每局游戏结束后提交分数 hash / 结果摘要
- [ ] 合约记录关键结果：玩家、分数、时间、赛季、任务 ID
- [ ] 前端即时展示“提交中 → 已确认 → 排名更新”状态

### D. 排行榜与赛季

- [ ] 实时排行榜：总积分榜 / 今日榜 / 小游戏榜
- [ ] 赛季快照：每周结束时保存 Top N 榜单 hash 或链上快照
- [ ] 徽章系统：Top 10、连续签到、完成 Monad 学习路径等
- [ ] 奖励结算：赛季结束后根据链上记录领取 NFT / token / 白名单资格

### E. AI Agent 互动扩展

- [ ] AI Agent 生成每日挑战题
- [ ] 用户完成挑战后，Agent 根据链上记录给出下一步任务
- [ ] Agent 读取用户公开积分 / 徽章，为用户生成个性化学习路径
- [ ] 关键任务完成证明上链，AI 推荐和解释保留在后端

### F. 合约最小设计

MVP 不需要复杂合约，最多 2-3 个：

1. `QuestRegistry`
   - 记录任务 ID、任务类型、积分、是否启用
   - 管理任务创建和关闭

2. `QuestProof`
   - 记录用户完成任务的证明
   - 防止同一任务重复 claim
   - 发出 `QuestCompleted(user, questId, points)` 事件

3. `BadgeNFT`（可选）
   - 给连续完成任务、赛季排名、特殊挑战发徽章 NFT

### G. 哪些放链上，哪些放数据库

| 数据 | 是否上链 | 原因 |
|------|----------|------|
| 钱包地址 | 是 | 用户公开身份和资产控制入口 |
| 关键任务完成记录 | 是 | 需要公开可验证，便于奖励结算 |
| 积分结算事件 | 是 | 排行榜和奖励需要可审计 |
| 赛季奖励领取 | 是 | 涉及资产或资格，应可验证 |
| 游戏过程每一帧 | 否 | 数据量大，没必要上链 |
| 临时 UI 状态 | 否 | 普通数据库 / local state 即可 |
| 推荐算法 | 否 | 不需要公开，且可能频繁变化 |
| 用户提交的截图原图 | 否 | 可存 IPFS / 对象存储，只把 hash 上链 |

## 3. Ops：3 个适合 Meme 或社区传播的产品切入点

### 切入点 1：Monad Speedrun Challenge

口号：
> “今天你上链了吗？0.4 秒一局，冲榜 Monad Speedrun。”

玩法：
- 每天发布一个极小任务：连接钱包、完成一次交互、回答一个 Monad 概念题、完成小游戏。
- 用户完成后获得积分和连续打卡 streak。
- 社区每天晒榜单、晒徽章、晒连续天数。

适合传播的原因：
- 门槛低，适合新人。
- 排行榜天然制造竞争。
- 可以和 Builder Camp 每日打卡结合。

### 切入点 2：Proof of Grind / 链上努力证明

口号：
> “不是嘴上说自己在学 Web3，而是每天留下 Proof of Grind。”

玩法：
- 用户把每日学习、任务完成、黑客松进度变成链上徽章。
- 每周生成一个“我的 Builder 进度卡片”。
- 可分享到 X / 微信群 / Discord。

适合传播的原因：
- 和新人学习痛点强相关。
- 将“坚持学习”变成可展示身份。
- 很适合做 Meme 图：钱包地址 + 等级 + streak + 本周称号。

### 切入点 3：AI Agent Dungeon on Monad

口号：
> “让 AI Agent 每天给你发一个链上任务。”

玩法：
- AI Agent 每天生成一个小挑战：学习题、交互任务、小游戏、生态探索。
- 用户完成后，Agent 根据链上记录升级用户角色。
- 社区可以晒自己的 Agent 评价，例如“今日称号：Gas Ninja / Quest Goblin / Monad Intern”。

适合传播的原因：
- AI Agent + Consumer Crypto 叙事强。
- 称号、角色、等级天然适合 Meme。
- 能把学习任务变得游戏化，而不是枯燥打卡。

## 4. 为什么不是普通数据库就够了？

这个问题要拆开看：

普通数据库足够处理：
- 页面展示
- 草稿
- 用户偏好
- 推荐算法
- 普通评论
- 游戏中间过程

但链上记录更适合处理：
- 用户是否真的完成关键任务
- 积分是否被平台随意修改
- 赛季奖励是否按规则发放
- 徽章是否可迁移到其他社区或应用
- 用户是否能把学习 / 贡献记录带到下一个项目

所以，这个产品的正确架构不是“所有东西都上链”，而是：

> 高频体验在前端和数据库完成，关键证明和奖励结算上 Monad。

这样 Monad 提供的是可信结算层和可组合身份层，而不是替代所有后端数据库。
