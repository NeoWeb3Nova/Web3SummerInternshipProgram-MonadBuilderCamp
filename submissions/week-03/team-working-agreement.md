## 1. 为什么需要这份协议

好的组队不是「兴趣一样」，而是「能一起把事情做出来」。  
本协议约定：**同步节奏、任务可见、决策方式、缺席处理、安全边界**，减少「以为对方知道」。

---

## 2. 每日同步

| 项 | 约定 |
|----|------|
| **时间** | 默认 **21:00–21:15 UTC+8**（15 分钟，可经全员同意改点） |
| **形式** | 微信语音 / 文字 stand-up；没空则 **21:00 前发异步更新** |
| **三句结构** | ① 昨天完成 ② 今天计划 ③ 卡点 / 需要谁帮 |
| **纪要** | PM 在看板或群里留 3 条 bullet（可 @ 责任人） |
| **迟到** | 超过 5 分钟不等人；缺席者事后读纪要并在看板更新状态 |

### 异步更新模板（复制即用）

```text
【Stand-up】姓名 / 日期
- 完成：
- 今天：
- 卡点：
- 需要：
```

## 3. 任务看板

| 项 | 约定 |
|----|------|
| **工具** | PM 在 Day 1–2 选定其一：**GitHub Projects** 或 **Notion**，选定后不换 |
| **列** | `Backlog` → `This Week` → `Doing` → `Review` → `Done` |
| **卡片最小字段** | 标题 · 负责人 · 截止日期 · 验收标准（一句话） · 关联文件/PR |
| **WIP 限制** | 每人 `Doing` 最多 **2** 张，防止伪并行 |
| **完成定义** | 有可检查产出（链接 / 截图 / PR / 录屏），不是「差不多了」 |
| **每日必更** | 下班或 stand-up 前，把自己的卡挪到正确列 |

### 角色默认看板责任

| 角色 | 主要卡片类型 |
|------|----------------|
| PM | 范围、用户问题、反馈收集、Pitch 结构、日程 |
| Dev | 链路实现、合约/前端、测试、部署、Evidence |
| UI | 流程线框、关键屏、确认页文案层级、Demo 视觉 |

## 4. 决策方式

| 类型 | 谁拍板 | 说明 |
|------|--------|------|
| **产品范围 / 本周砍什么** | PM 主导，Dev 评估可行性，UI 评估可演示性 | 争议超过 15 分钟 → 记两案，**当晚 stand-up 二选一** |
| **技术选型 / 实现路径** | Dev | 影响 Demo 安全或时间的选型需在群里说一句 |
| **界面与演示路径** | UI | 与用户流程冲突时 PM 仲裁 |
| **安全边界** | **不可投票否决**（见第 6 节） | 任何成员可一票否决「跳过确认 / 自动签名」类方案 |
| **对外 Pitch 口径** | PM 统稿，Dev/UI 各审一段 | 对外数字与链接以 Evidence 为准 |

### 决策记录

- 重要决定（锁定核心动作、砍功能、改方向）由 **PM 写一行** 到群公告或 `submissions/week-03/` 对应文档。  
- 格式：`[日期] 决定：…｜原因：…｜反对意见：无 / …`

## 5. 缺席与沟通边界

| 情况 | 处理 |
|------|------|
| 预计缺席 stand-up | 提前在群里说 + 发异步三句 |
| 当天无法投入 | 中午前告知 PM，PM 重排优先级，不默认「别人顶上」 |
| 连续 2 天失联 | PM 私聊确认；仍无回应则缩小该角色 scope，保证 Demo 仍可展示 |
| 冲突 / 情绪 | 先私聊 1:1，再拉第三人；不在公开社群甩锅 |
| 回复预期 | 工作时段 **12 小时内** 有回复（已读可先回「收到，晚些处理」） |

---

## 6. 安全边界（硬约束）

以下为 **不可妥协** 项，与 Idea Pitch / Moss 定位一致：

1. **不托管用户私钥**；不在 Demo 中要求明文助记词。  
2. **Moss 路径：只规划与模拟，不自动签名、不自动广播**。  
3. 涉及资产变动的操作，UI 必须有 **明确确认步**（展示 Plan / 关键 effects / 风险提示）。  
4. 测试网与主网地址、交易哈希必须 **可核验**，禁止编造 Evidence。  
5. AI 生成的代码 / 文案须经 **负责人人工过目** 再进 Demo 或对外材料。  
6. 用户测试数据（微信、反馈原话）不外传出队外。

若有人提出违反以上条款的「更快 demo」方案 → **当场否决**，改用 Mock / 模拟数据。

## 7. 质量与 Demo Evidence

| 角色 | 本周最低 Evidence |
|------|-------------------|
| Dev | 可运行入口（Repo/CLI/网页）+ 至少一种链上或本地验证记录 |
| UI | 主路径 3–5 屏可展示（设计稿或实现）+ Demo 录屏可用画面 |
| PM | Problem/Scope 文档齐 · 反馈记录 · Pitch 提纲 |

共享原则：

- **小步提交**：一天至少一次可见进度（PR / 文档 / 截图）。  
- **先能演示，再能好看**；UI 在链路未通前可用静态 Mock。  
- Review：合并前至少另一人扫一眼（文字「LGTM」即可）。
