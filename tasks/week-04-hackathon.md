# Week 4: Monad Builder Hackathon

## 本周目标

把 Week 3 的 Mini Demo 打磨成 Hackathon 参赛作品，完成提交和 Demo Day 展示。

开营仪式强调：Week 4 是“战场”，围绕 idea brainstorm、团队组建、产品冲刺、导师辅导、用户反馈和 Demo Day 展开。优秀项目有机会获得奖金、孵化、生态曝光和后续连接。

## 本周原则

- 只保留最能证明价值的核心功能
- Demo 必须稳定，宁可小而完整，不要大而不可运行
- 每个提交材料都要能回答：为什么是 Monad？
- 记录用户反馈和导师反馈，作为作品可信度
- 提交前确保 Repo 可复现、README 清楚、链接可访问

## Day-by-Day 任务

### Day 22: Hackathon 定题与计划

- [ ] 确认 Hackathon 赛道、要求、评分标准和截止时间
- [ ] 明确最终提交项目名和一句话价值主张
- [ ] 确定本周必须完成 / 可选 / 不做事项
- [ ] 制定每日 standup 时间和负责人
- [ ] 输出 `submissions/week-04-hackathon/plan.md`

### Day 23: 核心功能冲刺

- [ ] 完成 MVP 核心功能
- [ ] 前端、合约、数据或后端完成主路径联调
- [ ] 开始写 README
- [ ] 记录关键技术决策

### Day 24: 产品体验与错误处理

- [ ] 优化 UI/UX 和用户引导
- [ ] 添加 loading / success / error 状态
- [ ] 检查钱包、网络、交易失败等边界情况
- [ ] 找至少 1 位外部同学试用

### Day 25: 测试、部署与验证

- [ ] 全流程测试
- [ ] 部署到 Monad Testnet
- [ ] 记录合约地址、交易哈希、Explorer 链接
- [ ] 部署前端 Demo
- [ ] 输出 `submissions/week-04-hackathon/deployment.md`

### Day 26: Pitch Deck 与 Demo Video

- [ ] 制作 Pitch Deck
- [ ] 录制 2-3 分钟 Demo Video
- [ ] 准备 Final Presentation
- [ ] 明确演示脚本：问题 → 解决方案 → Demo → Monad 价值 → 下一步

### Day 27: 提交前检查

- [ ] 核对 README 清单
- [ ] 确保 Repo 可复现
- [ ] 检查所有链接可访问
- [ ] 检查合约安全性和测试网配置
- [ ] 预演 Demo Day 展示

### Day 28: 提交与 Demo Day

- [ ] 正式提交 Hackathon 作品
- [ ] 参与 Demo Day
- [ ] 记录评委/导师/同学反馈
- [ ] 写 `submissions/week-04-hackathon/retro.md`
