# Week 3: 合作实战期 / Mini Demo

## 本周目标

跨轨组成小组，把 Week 1-2 的学习、调研和技术基础推进成一个可展示的 prototype。

开营仪式强调：AI 可以快速生成代码，但真实产品仍需要产品判断、用户痛点、市场反馈和跨角色协作。因此 Week 3 的重点不是“堆功能”，而是找到一个真实问题，并让 Tech / Ops / Research 一起把它推进到 Mini Demo。

## 本周原则

- 小团队优先跨轨组合：Tech + Ops + Research
- 问题必须真实，不只为写代码而写代码
- 先定义用户和痛点，再做功能
- 用 AI 提效，但由人判断范围、优先级和质量
- 每天保留反馈和迭代记录

## Day-by-Day 任务

### Day 15: 组队与角色确认

- [ ] 找到至少 2 名不同 Track 的伙伴
- [ ] 确定团队角色：Tech / Ops / Research / Pitch
- [ ] 对齐每个人本周可投入时间
- [ ] 在 `submissions/week-03/team.md` 记录团队信息和分工

### Day 16: 问题定义

- [ ] 从 Week 2 的用户观察/研究/技术实验中选 2-3 个候选问题
- [ ] 用“谁在什么场景下遇到什么问题”写清楚用户痛点
- [ ] 验证问题是否适合 Monad / EVM / 链上场景
- [ ] 输出 `submissions/week-03/problem-statement.md`

### Day 17: 方案设计与范围裁剪

- [ ] 设计最小可行方案
- [ ] 明确本周不做什么
- [ ] 画出用户流程 / 系统流程
- [ ] 确定链上部分、前端部分、运营/研究部分
- [ ] 输出 `submissions/week-03/solution-design.md`

### Day 18: MVP 搭建

- [ ] Tech: 搭建核心合约与前端框架
- [ ] Ops: 准备目标用户反馈问题、内容草稿或测试邀请
- [ ] Research: 补充竞品/生态/技术论证
- [ ] 每个角色记录当天产出和卡点

### Day 19: 集成与用户反馈

- [ ] 集成核心流程
- [ ] 找 1-3 位同学或助教试用 / 看 demo
- [ ] 记录反馈：不理解什么、卡在哪里、是否愿意继续用
- [ ] 修复最影响展示的问题

### Day 20: Mini Demo 准备

- [ ] 制作 3-5 分钟演示流程
- [ ] 准备一句话价值主张
- [ ] 准备 Demo 脚本：问题 → 方案 → 演示 → 为什么 Monad → 下一步
- [ ] 预演并收集反馈

### Day 21: Mini Demo 与复盘

- [ ] 参与 Mini Demo / 小组展示
- [ ] 记录导师/同学反馈
- [ ] 判断是否值得进入 Week 4 Hackathon 冲刺
- [ ] 输出 `submissions/week-03/retro.md`
