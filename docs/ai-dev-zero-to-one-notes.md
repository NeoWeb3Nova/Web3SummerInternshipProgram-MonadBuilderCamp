## 一、什么是「从 0 到 1」

### 1.1 「0 → 1」：一个可验证核心价值的 Demo

- 目标：做出一个能让目标用户（C 端用户或投资人）体验并判断价值的 Demo。
- 需要清晰传达：
  - 产品解决什么问题
  - 提供了哪些功能
  - 核心流程是怎样的

### 1.2 「1 → 100」：从 Demo 变成可持续的生产系统

- 把可验证的 Demo 扩展为可持续迭代的系统。
- 在可靠性、安全性上有一定基础，不再是临时验证性的产品。
- 后续优化体验、商业模型等都在这个阶段完成。

### 1.3 AI 出现前后的变化

- AI 以前：零到一门槛高，往往需要找到认可想法的合伙人（前后端、APP 等）一起开发。
- AI 以后：个人可以独立完成 Demo，成本更低、速度更快。

## 二、AI 开发的两大优势

### 2.1 降低门槛

- 即使没有编程经验，也能通过清晰描述需求，让 AI 快速生成可运行的代码。
- 不懂代码也能跑起来，生成可以展示的东西。

### 2.2 提升效率

- 功能迭代快、缺陷反馈快。
- **最重要的一点：大大缩短了重构周期**。
  - 以前重构成本高昂，可能导致业务停滞。
  - 现在 AI 让重构变得便宜，甚至成为必要环节——因为 AI 生成代码快、跑偏概率高，需要通过不断重构让系统模块化、边界清晰、更可控。

## 三、AI Agent：从回答问题到执行任务

### 3.1 从「更高效的 Google」到「助手」

- 以前：LLM 主要是问答式工具，帮助快速获取答案。
- 现在：Agent 能够执行任务——接收目标、分析步骤、感知环境、规划行动、调用工具、返回结果。
- 核心：把推理转化为「受约束、可观察、可验证」的行动。

### 3.2 能力飞速提升皂四个原因

| 因素 | 说明 |
|---|---|
| 模型更强 | 推理能力「脑子越来越好用」 |
| 上下文更长 | 能读取更多代码、文档、历史与中间结果，做出更精准判断 |
| 工具丰富 | CLI 工具、MCP、Bash 执行、API 调用等让 Agent 工具箱越来越强 |
| 闭环更完整 | Memory 持久化 + 在交互中学习用户偏好，甚至形成 Skill 进行自我进化 |

### 3.3 Agent 的本质

- 不断拓展自己的能力边界，像前辈说的「不要局限自己」。

## 四、AI 开发实践：必须理解的几件事

### 4.1 Agent / AI Coding 配置文件（以 `.cursorrules` / `.claude.md` / `AGENT.md` 等为代表）

- **全局级配置**：位于 Agent 工具自身的配置目录，定义个人偏好、工作风格、长期规则。
  - 例：Codex 的 `~/.codex/` 、Claude Code 的 `~/.claude/`。
- **项目级配置**：在项目根目录下写入，约束：
  - 架构边界
  - 编码规范
  - 允许/禁止做的事
  - 交付标准
  - 评审需求
  - CI/CD 流水线
- 为什么重要：
  - 所有 AI 工具都会阅读这个项目级配置。
  - 对于开源项目，这个文件有时比 README 更能说清楚架构和开发入手方式。

### 4.2 TDD（测试驱动开发）

- 流程：先写测试 → 定义期望输出 → 测试失败 → 最小实现 → 测试通过 → 重构。
- 为什么适合 AI 编程：
  - 把正确实现转化为可执行的验收标准。
  - 让 Agent 在生成代码时快速自我纠正。
  - 防止修复 A 问题时破坏 B 功能（即使破坏也能快速发现）。
- 实际体验：例如 Rust 开发时，Agent 会主动运行 `cargo test`，测试用例越全面，它越能在开发过程中及时发现并修正问题。

### 4.3 SDD（规范驱动开发 / Spec-Driven Development）

- 核心目标：**先对齐意图，再生成代码**。
- 步骤：
  1. 为什么要做？
  2. 要做成什么样？
  3. 怎么实现？
  4. 拆分任务、逐一实现
  5. 归档留痕，便于后续溯源
- 工具推荐：
  - **全新项目**：`Spec Kit`（更重、更规范、产出更多）。
  - **现有项目迭代**：`OpenSpec`（更轻量，聚焦当前需要实现的东西）。
  - 也可以团队维护自己的规则和模板。

### 4.4 跟着 AI 一起提升的四个能力

| 能力 | 要点 |
|---|---|
| 会描述 | 把想法写成目标，说清用户、场景、约束、验收标准，学会写好 Prompt |
| 会拆解 | 把复杂任务拆成可执行、可理解、可验证的小步骤，理清每步依赖与风险 |
| 会验证 | 通过手动测试、自动化测试、TDD、日志等方式参与验证，自己把关结果 |
| 会设计 | 在 AI 协作中逐步学会架构边界、数据流、协作规则与长期质量体系 |

### 4.5 一个有趣的现象：「拷贝出去的链接是 127.0.0.1」

- 这不是 AI 专有的问题：如果缺乏计算机网络知识，即使自巶学也可能犯同样的错误。
- AI 的作用是**加速了跳过知识学习过程的实现阶段**，但也让知识缺失更快暴露出来。
- 反而有了 AI 后，遇到这类问题可以更快地问、更快地纠正，不用翻书或问人。

## 五、Web3 开发的技术路径

### 5.1 区块链是什么

- **专业定义**：由多节点共同维护，通过密码学与共识机制验证状态转换的分布式账本（复合状态机）。
- **通俗理解**：**异步的、强一致性的、可编程金融平台**。
  - 异步：交互结果不是立即返回，需要等待打包、执行、共识完成。
  - 强一致性：所有节点对同一笔交易/状态执行出一致结果。
  - 可编程：通过智能合约自定义逻辑。
  - 金融平台：资产流通、DeFi 、借贷等都与金融密切相关。

### 5.2 以太坊为例：Web3 开发的五个方向

| 方向 | 主要工作 | 与 DApp 项目方的关系 |
|---|---|---|
| **DApp 开发** | 前端页面、钱包连接、UI 交互、业务逻辑 | 紧密，直接面向用户 |
| **智能合约开发** | 链上状态管理、资产管理、协议规则定义 | 紧寁耦合，开发 DApp 通常都要涉及 |
| **钱包开发** | 密钥管理、账户流式管理、签名体验、交易体验 | 较独立，项目方通常使用现成钱包（MetaMask、OKX Wallet 等） |
| **节点开发** | P2P 网络、执行层/共识层规则、存储、RPC | 较独立，通常由公链团队维护，DApp 项目方使用 RPC 接口 |
| **安全审计** | 代码 Review、漏洞发现、审计报告 | 最后一道红线，保障项目方和用户资金安全 |

### 5.3 关于「财务/会计」类比的现场互动

- 有人将五个方向类比为财务流程：DApp 是交付物、合约是合同、钱包是公账、节点是合同签约物件、审计是财务审计。
- 讲者观点：五个方向更像是「开发者可以深挖的五个站点」，不是严格的财务分工。
- 安全审计确实最贴合这个类比：是项目上线前最后一道安全保障。

### 5.4 同共需要学习的基础

- 密码学与密钥体系
- 哈希技术
- 以太坊基础（常见的开发入门方向）

## 六、实操启示：从 0 到 1 的行动 Checklist

1. **明确 Demo 的目标用户和核心流程**，不要一开始就想做「系统」。
2. **用 AI 快速搭建可运行的原型**，接受会有重构、跑偏的过程。
3. **在项目目录下写入 Agent/AI Coding 配置文件**，固化架构边界、编码规范与交付标准。
4. **用 TDD 和 SDD 作为开发节奏**，让 AI 有可验收的标准，避免无限迭代跑偏。
5. **选择 Web3 技术方向时先对齐公链基础**，以太坊依然是最友好的入门系统之一。
6. **不要「脑袋放空」式用 AI**：在协作中主动提升描述、拆解、验证、设计四项能力。
