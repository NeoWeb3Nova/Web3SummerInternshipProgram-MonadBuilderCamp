# 交付物标准

## 总体原则

所有交付物应满足：

- **公开可验证**: 尽量使用公开链接、链上地址、GitHub Repo、交易哈希
- **可复用**: 文档清晰，他人能看懂并复现
- **真实**: 记录真实过程，包括失败、卡点、修改和迭代
- **AI 协作透明**: 标注 AI 在每一步中的作用与人的判断
- **职业可展示**: 能进入作品集、简历、面试表达或社交媒体展示

## 每周交付层级

| 层级 | 目的 | 示例 |
|------|------|------|
| Proof of Learning | 证明学过 | 每日打卡、学习日志、术语笔记 |
| Proof of Work | 证明做过 | 钱包地址、交易哈希、合约地址、Repo、Demo |
| Proof of Collaboration | 证明协作过 | 团队分工、反馈记录、Demo Day 复盘 |
| Career Evidence | 证明能被雇佣/合作 | Portfolio、简历 bullet、面试讲稿、公开文章 |

## 最终交付物清单

### 1. 链上产品 Demo

- 可访问的 Demo 链接（Vercel / Netlify / 其他）
- 支持的链/网络（Monad Testnet 优先）
- 核心功能说明
- 测试账号/钱包说明
- 至少一条可验证交易或链上状态

### 2. GitHub Repo

- 清晰的 README
- 代码结构合理
- 包含部署脚本/说明
- 包含环境变量模板
- 包含本地运行步骤
- 开源协议
- Commit 历史反映真实开发过程

### 3. Build Log

- 记录 AI 使用场景
- 记录关键决策与技术选型
- 记录遇到的问题与解决方案
- 记录 prompt / 调试思路 / 人的判断
- 每日或每周更新

### 4. Track-specific 材料

根据所选 Track：

- **Tech**: 部署文档 + 技术复盘 + 合约/前端测试记录
- **Ops**: Campaign 方案 + 增长数据 + 用户反馈 + 复盘
- **Research**: 研究文章 + 生态地图 + Product Thesis + 引用来源

### 5. Hackathon 提交包

- Pitch Deck
- Demo Video（建议 2-3 分钟）
- Final Presentation
- 团队介绍与分工
- Demo 链接
- Repo 链接
- 合约地址 / 交易哈希 / Block Explorer 链接

### 6. Portfolio Page

- 一页式项目展示
- 包含项目背景、用户问题、个人角色、成果、链接
- 说明 AI 协作方式
- 适配简历与面试场景

## 面试表达检查清单

回答以下问题：

1. 为什么选择 Monad？
2. 你在项目中负责什么？
3. 你如何验证这是一个真实问题？
4. 你如何使用 AI？哪些判断是你自己做的？
5. 遇到的最大困难是什么？如何解决？
6. 如果再做一次，你会怎么改进？
7. 这个项目如何继续做成真实产品？
