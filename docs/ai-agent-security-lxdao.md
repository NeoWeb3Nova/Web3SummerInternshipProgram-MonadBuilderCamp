- 主讲人：GoPlus 阿路，负责 Web3 + AI 安全研究。

## 2. 主要观点：AI Agent 时代的“潜多拉魔盒”

>无法审计的自主性。

- Agent 被授予钱包权限、API 密钥、系统命令权限后，一旦失控就可能成为“最危险的内鬼”。
- 常见失控表现：
  - 幻觉 / 偏离目标
  - 不听指令 / “理解过度”
  - 关键操作忘记二次确认
  - 多 Agent 协作时上下文偏移，难以实时防护

## 3. 四类高风险实例

### 3.1 过度授权

- **核心问题**：Agent 要帮人干活，就必须授权；但很多场景下授权过宽。
- **加固建议**：
  - 最小权限原则：只给完成任务所需的最低权限。
  - 定期审查和变更权限，需求变化后及时收回。
  - 资金转账、邮件删除等敏感操作必须二次确认。

### 3.2 恶意 Scale / 插件（供应链）

- **扫描结果**：针对某 Scale 市场 Top 100 插件扫描，约 11% 存在严重风险、高风险占 10%，仅 62% 为低风险。
- **典型案例 1：2025 年 2 月 300+ 恶意 Scale 混入市场**
  - 攻击目标：链上交易工具（Pump.fun / Bebop 等）。
  - 恶意行为：窃取钱包私钥、交易记录、交易所 API Key、LLM API Key、SSH 密钥等凭证；有的直接安装木马接管设备。
- **典型案例 2：Scale 市场排名第一的木马**
  - 通过恶意刷量/虚假宣传冲到榜首，用户下载后被窃取密钥。
- **加固建议**：
  - 从官方渠道下载插件，安装前/后做安全扫描。
  - 关注插件权限申请，避免过度授权。

### 3.3 Prompt 注入（提示词注入）

- **攻击原理**：攻击者在网页、文档、评论中嵌入隐藏恶意指令，Agent 读取后执行。
- **典型案例 1：伪装 Base 链教程的转账指令**
  - 在链上教程中埋入 Prompt，诱导 Agent 向某地址转账 0.1 ETH。
- **典型案例 2：GitHub Issue 中的 NPM 恶意仓库指令**
  - 攻击者通过 GitHub Issue 提交包含恶意代码的指令，诱导 Agent 安装后门包。
- **典型案例 3：Grok / ai16z 摩斯密码事件**
  - 攻击者在 X 评论中用摩斯密码发送转账指令，Agent 解码后执行了转账，导致资产损失。
  - 这一事件还涉及“过度授权”：Worker Bot 采用 NFT 持有即授权机制，任何评论都会立即执行，缺乏二次验证。
- **加固建议**：
  - 对外部输入做清洗和过滤，特别是不可见文本、编码内容。
  - 敏感操作不要仅依赖 LLM 的“理解”，设置确认阶梯。
  - 更系统的案例与防御思路详见下文「高风险实例——Prompt 注入」章节。

### 3.4 多 Agent 协作与上下文偏移

- **高风险场景**：
  - 用户把 Agent 拉进群聊，群友可以直接索要密钥、私钥或编写搜索密码文件的代码。
  - 多 Agent 协作时，某个 Agent 可能产生幻觉或忘记二次确认条件，直接执行高危操作。
- **加固建议**：
  - 不要把存在高权限的 Agent 拉入公开群聊。
  - 采用“审核 Agent”设计：主 Agent 只负责执行，独立审核 Agent 专门判断是否达成目标、是否该执行。

## 4. 安全防线构建：Agent Guard 的思路

### 4.1 产品定位

- **产品定位**：面向原生 AI Agent 系统的端到端安全平台，不仅保护单个 Agent，而是覆盖编排模型、MCP/工具服务、多 Agent 集群。
- **闭环**：
  - 发现（资产梳理、风险识别）
  - Red Team 验证（模拟攻击、渗透测试）
  - Runtime 防护（实时拦截高危命令、网络请求评估）
  - 治理（策略持续优化、事件追溯）

### 4.2 关键能力

| 能力 | 说明 |
|---|---|
| 实时动态防护 | 在 Agent 执行命令/发起网络请求前实时评估，拦截 `rm -rf`、凭证窃取、越权数据外发、删除邮件等。 |
| 恶意插件扫描 | 安装前/后通过 hook 触发扫描，发现后门、恶意代码。 |
| 风险情报 | 覆盖 Scale 插件、MCP、供应链、钓鱼网站、恶意输入。 |
| 自定义策略 | 支持黑白名单、预设+自定义策略，减少误报/漏报。 |
| 事件记录 | 高危命令触发后保留记录，便于溯源。 |

### 4.3 基础安全加固建议

**核心思路：最小权限控制 + 人工审批机制 + 实时安全监控**

三者共同构成 AI Agent 的基础安全防线。

#### 1. 遵循最小权限原则

- 仅授予 AI Agent 完成当前任务所必需的最低权限。
- 定期审查权限清单，及时撤销不再需要的访问权限。

#### 2. 引入 Human-in-the-Loop（HITL）

为高风险操作设置人工介入和安全基线，例如：

- 资金转账
- 邮件删除
- 数据修改
- 权限变更
- 对外发布

此类敏感操作应要求人工复核或二次确认，避免 Agent 自主执行不可逆操作。

#### 3. 部署安全监控工具

使用 AgentGuard 等专业安全工具，对 Agent 的运行过程进行实时监控，重点识别：

- 异常调用行为
- 越权访问
- 敏感信息泄露
- 可疑工具调用
- 高风险操作
- 与正常行为基线不一致的活动

## 5. AgentGuard 详解（实现参考）

### 5.1 产品定位

AgentGuard 是面向 AI Agent 系统的端到端安全平台。它保护的对象不是孤立的单个 Agent，而是整个 Agent 系统，包括：

- Agent 本身
- Agent 编排的模型
- 工具
- MCP 服务
- RAG / 数据
- 组织内部的整个 Agent 集群

AgentGuard 围绕以下四个环节构建完整安全闭环：

```text
Discover（发现资产） → Red-Team（红队验证） → Defend（运行时防御） → Govern（统一治理）
```

目标是把「攻」与「防」打通，形成一条完整的 AI Agent 安全闭环。

### 5.2 四项核心能力

| 能力 | 做什么 | 关键产出 |
|---|---|---|
| **Discover（发现）** | 跨代码仓库、云/模型厂商、运行主机盘点 AI 资产；进行框架指纹识别；构建资产与攻击面图谱；进行供应链信誉评分 | 一份实时的 **AI 资产与风险清单** |
| **Red-Team（红队验证）** | 对已经发现的资产进行受控、`dry-run` 式红队模拟，包括提示词注入、工具审批绕过、RAG 泄露、MCP 提权、密钥暴露、Web3 合约风险等 | Findings，包括**严重程度、证据、责任人**，并形成“发现问题 → 修复 → 复测”的流程 |
| **Defend（运行时防御）** | 本地守卫在动作发生的当下进行判断：`allow`、`warn`、`approve`、`block`；同时提供钩子拦截、深度扫描和每日巡检 | **实时拦截 + 脱敏审计日志** |
| **Govern（治理）** | 通过一个统一控制面管理策略下发、审批工作流、审计时间线、威胁情报网络，以及 `deny-first` 企业治理 | 通过单一面板查看全部 **Agent 风险** |

### 5.3 整体工作流程

```text
Discover
发现有哪些 Agent、模型、工具、MCP 和数据资产
        ↓
Red-Team
主动模拟攻击，验证这些资产是否存在真实风险
        ↓
Defend
在 Agent 实际运行时拦截危险动作
        ↓
Govern
通过统一控制台管理策略、审批、审计和风险
        ↓
修复与复测，持续形成安全闭环
```

### 5.4 一句话概括

AgentGuard 不只是给单个 AI Agent 安装一个“安全插件”，而是对企业内部完整的 Agent 系统进行资产发现、攻击验证、实时防御和统一治理。

## 6. Q&A 精华

- **问题**：为什么 AI 会“跳过”或忘记应该做的事？  
  **回答**：LLM 是概率分布模型，不是精确程序。当没找到确切答案时会通过注意力机制去寻找周边答案，从而带偏整个思维链路；一旦偏了很难拉回来。
- **问题**：多 Agent 同时执行，有的忘记有的没忘，取多数决策是否可靠？  
  **回答**：在较差的基模环境下不靠谱。更好的做法是设计专门的“审核 Agent”：执行 Agent 带着精准上下文返回给审核 Agent，审核 Agent 只判断“是否做对”，没做对则打回重做。

# 高风险实例——Prompt 注入

> 核心逻辑：外部内容是数据，不是指令。一旦 Agent 把不可信文本误解为可信命令，就可能直接调用高权限工具，造成资产、权限或供应链损失。

## 一、攻击流程

```text
攻击者创建恶意内容
        ↓
植入隐藏恶意指令
        ↓
Agent 自动执行
```

## 二、攻击原理

攻击者在网页、文档、Issue、代码注释等内容中构造隐藏的恶意指令，诱导 AI Agent 对其进行直接或间接的解析与执行。

这类攻击的关键问题在于：

> Agent 不仅“读取”外部内容，还可能把外部内容错误地当作可信指令，并调用工具自动执行。

---

## 三、典型案例

### 典型案例 1：Moltbook 恶意 Submolt

攻击者 **@thechandog** 在 **#Moltbook** 上创建了 30 多个恶意 Submolt，将其伪装成 Base Chain L2 TypeScript 教程。

这些内容中包含多个 Prompt 注入 Payload，并植入了直接向链上地址：

```text
0x8eadc7cc0a77594e3fa999e80e1ccb7f4e1c04e0
```

转账 **0.1 ETH** 的恶意指令。

#### 攻击链路

```text
伪装成技术教程
    ↓
隐藏 Prompt 注入指令
    ↓
Agent 读取并解释内容
    ↓
诱导 Agent 发起链上转账
```

---

### 典型案例 2：Clinejection 供应链攻击

攻击者提交了一个植入 Prompt 注入指令的 GitHub Issue。

Cline 中的 Claude 在解析该 Issue 时，被诱导执行 `npm install`，安装了恶意仓库：

```text
github-actions/cline
```

恶意代码随后执行缓存投毒，并窃取了 npm token。

攻击者又将存在问题的：

```text
cline@2.3.0
```

发布到 npm，并在 CLI 中加入以下 `postinstall` 指令：

```json
{
  "postinstall": "npm install -g openclaw@latest"
}
```

最终导致超过 **4000 台用户设备**误装 OpenClaw。

#### 攻击链路

```text
恶意 GitHub Issue
    ↓
隐藏 Prompt 注入指令
    ↓
Claude/Cline 解析内容
    ↓
自动执行 npm install
    ↓
安装恶意依赖
    ↓
缓存投毒、窃取 npm token
    ↓
污染 npm 软件供应链
    ↓
大量用户设备误装软件
```

# 四、安全加固建议

## 1. 采用零信任原则

严防 Prompt 注入攻击，不应因为内容来自 GitHub、技术文档或开源仓库，就默认其中的指令可信。

**切勿让 Agent 盲目执行来历不明的代码、命令或操作指令。**

## 2. 引入 Human-in-the-Loop（HITL）

设置安全基线，对以下敏感操作要求人工复核或二次确认：

* 资金转账
* 邮件删除
* 安装软件包
* 执行 Shell 命令
* 修改系统文件
* 读取或上传密钥
* 发布 npm 包
* 调用高权限工具

## 3. 部署安全监控工具

使用 AgentGuard 等专业工具，实时监控 Agent 的异常行为，例如：

* 未经授权的命令执行
* 可疑依赖安装
* 敏感凭证访问
* 异常链上转账
* 工具调用越权
* Prompt 注入指令
* 软件供应链污染

## 五、核心结论

Prompt 注入并不只是“让 AI 说错话”，真正的风险是：

> **攻击者通过污染 Agent 读取的外部内容，把不可信数据变成可执行指令，进而控制 Agent 的工具、权限和资产。**

因此，Agent 系统必须严格区分：

```text
外部内容是数据，不是指令
```

并通过 **零信任、最小权限、人工审批、运行时监控**共同构建安全防线。

# 高风险实例——@Grok 钱包劫持

## 一、事件概述

Grok 在处理一段恶意 Prompt 时遭遇 **Prompt Injection（提示词注入）**。

攻击者利用经过精心设计的提示词，以及 Grok 钱包持有的 **Bankr Club Membership NFT** 所赋予的 Agent 操作权限，诱导 Agent 自动执行转账，最终导致约 **30 亿枚 $DRB** 被转出。

>
> 感谢你的完整拆解，提醒大家用 AI Agent 时要严管权限。教训已吸取，下次我更谨慎！😂

发布时间：

```text
2026 年 5 月 4 日 下午 6:14
5,095 次查看
```

## 三、Bankr 的授权机制

### 1. NFT 持有即授权

Bankr Club Membership NFT 的设计初衷是：

> **只要钱包持有该 NFT，就自动获得相应的 Agent 操作权限。**

当目标钱包——例如本次被攻击的 Grok 钱包——持有该 NFT 时，`@bankrbot` 会自动获得完整的 Agentic 权限，包括：

* `transfer`：转账
* `swap`：代币兑换
* `cross-chain bridging`：跨链桥接
* 其他链上资产操作

这属于典型的：

```text
On-chain ownership-based access control
基于链上所有权的访问控制
```

也就是说，系统通过“钱包是否持有指定 NFT”判断该钱包是否已经完成授权。

---

## 四、攻击过程

### 第一步：发布恶意 Prompt

攻击者在 X 平台公开发布经过特殊设计的恶意 Prompt。

### 第二步：诱导 Grok 解析

恶意 Prompt 并没有直接写出明显的“转账”命令，而是通过特定语义和复杂编码进行伪装。

例如利用 AI 对以下内容的解析能力：

* 摩斯密码
* 编码字符串
* 隐藏语义
* 分步解密任务
* 角色扮演指令

### 第三步：将转账包装成解密游戏

攻击者将真实的“转账”指令伪装成一个看似无害的“解密游戏”，让 Grok 将恶意内容解释为正常任务。

### 第四步：Bankr 验证 NFT 权限

Bankr 识别 Grok 解析出的 Prompt 指令，并检查 Grok 钱包是否持有 Bankr Club Membership NFT。

```text
检查 NFT
    ↓
确认钱包已获得授权
    ↓
允许 Agent 执行链上操作
```

### 第五步：自动签名并广播交易

NFT 权限验证通过后，Bankr 直接代表 Grok：

* 构造交易
* 签名交易
* 广播交易
* 转移钱包资产

最终，约 **30 亿枚 $DRB** 被转出。

## 五、完整攻击链路

```text
攻击者发布恶意 Prompt
        ↓
利用编码和语义隐藏真实意图
        ↓
将“资产转账”伪装成“解密游戏”
        ↓
Grok 解析并接受隐藏指令
        ↓
Bankr 检查 Grok 钱包持有的会员 NFT
        ↓
NFT 验证通过，确认 Agent 操作权限
        ↓
Bankr 代表 Grok 签名并广播交易
        ↓
钱包资产被转移
```

## 六、此次攻击的关键特点

### 1. 攻击者不需要获取私钥

整个过程中，攻击者并没有控制 Grok 钱包的私钥。

攻击者依赖的是：

* Grok 对恶意 Prompt 的解析
* Bankr NFT 所赋予的操作权限
* Agent 自动调用链上工具的能力

因此，这并不是传统意义上的“私钥被盗”，而是：

> **攻击者通过操控 Agent 的决策过程，借用 Agent 已经拥有的合法权限完成资产转移。**

### 2. 没有发送传统病毒或木马

攻击者并未安装病毒，也没有直接攻破系统。

真正的攻击载荷是一段自然语言和编码内容。

```text
恶意文本
    ↓
AI 解析
    ↓
形成操作指令
    ↓
调用高权限工具
```

### 3. 授权本身合法，执行意图恶意

从 Bankr 的权限系统来看：

* Grok 钱包确实持有 NFT
* NFT 确实赋予了相应权限
* Agent 确实有权执行转账

问题并非“没有授权”，而是系统没有确认：

> **这次操作究竟是不是钱包所有者的真实意图。**

---

# 七、暴露出的安全问题

## 1. NFT 持有即授权的权限过大

仅凭持有某个 NFT，就自动获得转账、兑换、跨链等完整权限，授权粒度过于宽泛。

系统没有进一步限制：

* 单笔转账金额
* 每日操作额度
* 允许操作的代币
* 允许交互的合约
* 允许接收资产的地址
* 权限有效时间
* 是否需要人工确认

## 2. Grok 对金融类 Prompt 缺少安全审查

涉及以下敏感动作的 Prompt，不应仅根据语义解析结果直接执行：

* 资产转账
* 代币兑换
* 跨链桥接
* 授权 `approve`
* 合约签名
* 私钥或凭证访问

## 3. 把“内容理解”直接连接到“资产执行”

该系统的危险结构在于：

```text
读取公开内容
        ↓
模型理解内容
        ↓
直接形成金融操作
        ↓
自动签名执行
```

中间缺少可靠的安全隔离层和人工审批环节。

---

# 八、安全加固建议

## 1. 降低 NFT 自动授权范围

NFT 不应直接赋予完整的钱包控制权，而应只作为身份或会员凭证。

更合理的权限模型是：

```text
持有 NFT
    ≠
自动获得无限资产操作权限
```

NFT 可以允许 Agent 访问服务，但转账、授权和跨链操作仍应单独验证。

## 2. 对金融类操作强制 Human-in-the-Loop

以下操作应要求用户人工确认：

* 转账
* Swap
* 跨链
* 授权代币额度
* 调用陌生合约
* 大额资产操作

确认页面至少要明确显示：

```text
转出资产：
转出数量：
接收地址：
目标网络：
调用合约：
预计手续费：
风险等级：
```

## 3. 设置限额和白名单

为 Agent 钱包设置：

* 单笔限额
* 每日限额
* 资产白名单
* 地址白名单
* 合约白名单
* 链白名单
* 操作频率限制

## 4. 对外部内容采用零信任策略

来自以下来源的内容一律应视为不可信数据：

* X 帖子
* GitHub Issue
* 网页
* PDF
* 邮件
* RAG 知识库
* MCP 返回结果
* 用户上传文件

基本原则是：

> **外部内容可以被读取和分析，但不能自动升级为高权限执行指令。**

## 5. 增加 Prompt Injection 检测

Agent 在执行前应检测：

* 隐藏指令
* 编码后的命令
* 角色覆盖
* 系统提示词绕过
* “忽略之前指令”
* 将危险操作包装成游戏或测试
* 分阶段拼接的恶意意图

---

## 九、核心结论

这起事件揭示了一种新型的钱包劫持方式：

> **攻击者不需要窃取私钥，而是通过 Prompt Injection 控制拥有钱包权限的 AI Agent，让 Agent 使用合法授权执行恶意交易。**

真正需要防范的，不只是传统的钱包密钥泄露，还包括：

```text
Prompt 被操控
        +
Agent 权限过大
        +
缺少人工审批
        =
钱包资产被合法权限“劫持”
```

```text
外部内容 → 被误解为指令 → 调用高权限工具 → 资产/供应链损失
```

## 十、总结

Prompt 注入的风险已经远超“让 AI 说错话”。当 Agent 被授予钱包、系统命令、软件供应链等高权限后，一段精心设计的外部文本就可能成为实际的攻击载荷。

有效的防御必须将 **零信任** 作为前提，把外部内容始终当作数据；通过 **最小权限** 限制 NFT/插件/工具的默认能力；通过 **Human-in-the-Loop** 为金融/高风险操作加上确认阶梯；通过 **运行时监控** 及时发现异常行为。

只有这四者结合，才能在 AI Agent 时代有效守住资产与系统的安全边界。
