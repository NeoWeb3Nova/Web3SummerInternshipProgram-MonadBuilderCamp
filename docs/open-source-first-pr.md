## 1. 一句话总结

Web3 开源贡献的门槛不是“先成为资深工程师”，而是先成为真实用户，再从一个已对齐的 Issue 出发，用一个只做一件事的小 PR、可复验的测试和清晰沟通，为项目降低维护成本。

---

## 2. Web3 职业信号：作品大于简历

分享前半段回顾了 Research、Ops 和 Tech 三条 Builder Track：

| 方向 | 核心工作 | 典型公开产出 |
|---|---|---|
| Research | 阅读协议、EIP/MIP、竞品和链上数据，形成分析 | 研究报告、协议 Review、数据分析 |
| Ops | 社区、内容、增长、品牌、BD、DevRel | Campaign、增长数据、内容和活动复盘 |
| Tech | 合约、前后端、AI Agent、安全与 Demo | GitHub 仓库、PR、可运行 Demo、合约 |

在 Web3 中，公开可验证的 Proof of Work 往往比包装后的简历更有说服力：

- GitHub 提交和 PR；
- 黑客松作品；
- 可运行 Demo；
- 协议研究；
- X / LinkedIn 上持续输出的高质量内容；
- 对生态项目的长期贡献。

因此，第一个 PR 的意义不只是“改了一段代码”，而是开始建立公开职业轨迹。

## 3. Moss 是什么

Moss 是面向 AI Agent 与链上协议交互的开源中间层。它希望降低 Agent 构造和验证链上交易的复杂度，并把签名前验证纳入默认流程。

### 四步工作流

```text
discover
发现协议及其能力
    ↓
load
加载协议动作、参数和约束
    ↓
action / build
构造未签名交易
    ↓
simulate
模拟执行，返回 effects 与风险信息
```

Moss 不负责替用户签名。它产生未签名交易并做模拟，让 Agent 或钱包在最后执行前判断：

- 交易实际做了什么；
- 得到了什么结果；
- 是否符合用户意图；
- 是否存在 warning 或异常 effects。

## 4. 贡献前先做用户

分享强调：不要一上来就让 AI Agent 改代码。

正确顺序是：

1. Clone 仓库；
2. 阅读 README / Getting Started；
3. 跑通 Example；
4. 在本地 fork 环境模拟一笔交易；
5. 阅读 simulate 输出；
6. 再决定哪里值得改。

只有用过产品，才能区分：

- 真问题与想象中的问题；
- 使用问题与框架问题；
- 局部修复与公共接口设计；
- “代码能跑”和“产品行为正确”。

## 5. 从 Issue 到第一个 PR

### 5.1 先找一个已存在的 Issue

优先选择：

- 描述清楚；
- 范围小；
- 与已有模板相似；
- 能在短时间内验证；
- Maintainer 明确欢迎贡献。

如果需求不清楚，先在 Issue 中提问或复述理解，不要直接提交大改动。

### 5.2 阅读项目的设计约束

Moss 仓库中的重要上下文包括：

- `packages/` 中已有协议包与模板；
- ADR：记录公共设计决策；
- `CLAUDE.md` / `AGENTS.md`：为 AI Coding Agent 提供仓库级规则；
- Test / Lint / Build：最终质量门槛。

AI Agent 读过文档，不代表贡献者可以不读。提交者仍然要理解变更内容并承担责任。

### 5.3 一个 PR 只做一件事

如果实现过程中发现无关问题：

- 当前 PR 保持原范围；
- 另开 Issue；
- 必要时单独提交第二个 PR。

小 PR 的价值是：

- 容易审查；
- 容易测试；
- 容易回滚；
- 容易判断是否解决原 Issue；
- 不会把设计争议和简单修复绑在一起。

### 5.4 用 TDD 和测试证明变更

推荐流程：

```text
阅读 Issue
  ↓
阅读 ADR / 模板 / 相邻实现
  ↓
明确验收条件
  ↓
先写或确认失败测试
  ↓
完成最小实现
  ↓
运行 focused tests
  ↓
运行 lint / build / 全量校验
  ↓
人工审计 AI 生成的 diff
  ↓
提交 PR
```

测试不是“最后补一下”，而是让贡献范围和正确行为变得可讨论。

## 6. AI Agent 可以参与，但不能替你负责

分享欢迎使用 AI Agent 开发，因为这也是检验贡献者是否会使用现代工具的一种方式。但有三个底线：

1. **先给上下文**：让 Agent 阅读 ADR、模板和仓库规则；
2. **跑完校验**：不能只看 Agent 声称“已完成”；
3. **人工审计**：提交者必须检查最终 diff、测试和影响范围。

### 推荐 Prompt 结构

```text
目标：完成 Issue X 中的单一需求
约束：遵循 ADR、CLAUDE.md / AGENTS.md 和现有模板
范围：只修改必要文件，不做无关重构
验证：先补测试，再运行 focused tests、lint、build
交付：解释每个文件的变化、风险和未解决项
```

### 常见失败

- 让 Agent 未读仓库就直接生成实现；
- PR 混入格式化、重构和无关修复；
- 只验证编译，不验证行为；
- 测试失败仍提交；
- 贡献者无法解释 Agent 写了什么。

## 7. Moss 的六类贡献方向

1. **主流协议 Adapter**：补充生态中重要协议；
2. **新兴协议接入**：持续支持新协议和长期 Bounty；
3. **基于 Moss 构建应用**：Agent 交易、收益策略、协议组合等；
4. **优秀 Demo 与案例**：形成可展示的真实用例；
5. **内容创作**：文章、视频、教程和社交媒体传播；
6. **社区协作**：开发、运营、研究共同完成产品与传播。

这意味着开源贡献不限于写代码。发现问题、补文档、解释设计、制作案例、帮助传播、组织协作，都是降低项目采用成本的贡献。

---

## 8. 团队协作比闭门造车更重要

分享鼓励学员在社区中公开需求，例如：

- 产品团队需要某个协议 Adapter；
- 开发者已做出 Demo，需要内容和运营伙伴；
- Researcher 可以比较协议能力和风险；
- Ops 可以把技术成果转成教程和传播素材。

团队协作的关键不是“所有人改同一段代码”，而是把需求、实现、验证和传播形成闭环。

## 9. 三种本周可执行贡献

### 动手：认领 Issue / 提交 PR

- 选择一个合理 Issue；
- 先沟通范围；
- 用小 PR 完成；
- 附测试与验证日志。

### 动笔：解释和传播

- 写项目介绍；
- 拆解核心设计；
- 制作入门教程或视频；
- 记录真实数据，如阅读量、使用量和转化。

### 搭 Demo：验证应用场景

- 基于 Moss 构建 AI × Web3 小应用；
- 先从本地模拟和无私钥流程开始；
- 明确 Demo 证明了什么、没有证明什么。

## 10. 如何判断“第一个 PR”选得好不好

一个好的首个 PR 通常满足：

- [ ] 问题由真实使用触发；
- [ ] 已与 Maintainer 对齐；
- [ ] 范围可以一句话说清；
- [ ] 有现有实现可参考；
- [ ] 有明确验收条件；
- [ ] 可以通过自动化测试验证；
- [ ] 不依赖大规模架构重写；
- [ ] 提交者能解释全部 diff；
- [ ] PR 描述包含问题、方案、验证和边界。

## 11. Q&A 精华

### 11.1 Monad 节点和开发工具是否开源？

分享提到 Monad / Category Labs 有多个公开仓库，覆盖开发者模板、客户端/共识相关实现和本地开发工具。普通应用开发者通常不必先运行完整本地共识集群，可以使用支持 Monad 特性的 Foundry 工具与 fork 测试完成大部分开发验证。

> 节点组件的具体仓库名在 ASR 中识别不稳定，应以 Category Labs / Monad 官方 GitHub 为准。

### 11.2 提交一个 PR 就算主力贡献者吗？

不算。单次 PR 是参与记录；主力贡献者意味着持续、有效地解决问题，参与讨论、维护和生态协作。分享提到持续贡献者可能获得 Bounty、活动支持和生态机会，但不应把奖励当作唯一目标。

### 11.3 开源后如何保护知识产权？

分享给出的思路包括：

- 选择合适的开源许可；
- 必要时采用延迟商业授权或部分开源；
- 用品牌、社区、生态关系和执行速度建立护城河；
- 对代码侵权可依许可证和社区机制沟通；
- 单纯 Idea 很难被保护，专利也通常不保护抽象想法本身。

更重要的判断是：如果一个 Idea 能被极低成本完整复制，那么真正缺少的可能是技术、数据、网络效应、渠道或迭代速度等护城河。

### 11.4 开源会不会反而扩大影响力？

会。Uniswap 等协议通过公开实现建立了事实标准。被 fork 不一定只意味着损失，也可能扩大技术范式、品牌和生态影响力。开源的保护不是“没人能复制”，而是让原创来源、迭代能力和社区共识持续可见。

## 12. 给当前项目的落地建议

当前仓库已经完成 Moss 阅读、Issue 分析、贡献计划和本地示例验证。下一步最符合本次分享的方法是：

1. 继续以 Issue #13 为唯一需求入口；
2. 在 Issue 中先对齐 ERC-4626 ABI 基础范围；
3. 阅读相关 ADR 与模板；
4. 把公共接口争议留在讨论中，不在首个 PR 一次解决；
5. 用 focused tests、build、lint 和变更日志证明实现；
6. 提交一个只做一件事的 Draft PR；
7. 根据 Maintainer Review 继续迭代。

这比“让 AI 一次生成完整 Adapter”更接近真实开源贡献。

## 13. 行动 Checklist

- [ ] 先跑通项目的最小示例。
- [ ] 阅读 README、CONTRIBUTING、ADR 和 Agent 规则。
- [ ] 从一个真实 Issue 开始。
- [ ] 在编码前复述需求与验收条件。
- [ ] 一个 PR 只做一件事。
- [ ] 先写或确认测试，再写实现。
- [ ] 运行 focused tests、lint、build 和必要的全量测试。
- [ ] 人工审计 AI Agent 的全部改动。
- [ ] PR 描述附真实命令与输出。
- [ ] 主动回应 Review，不把修改意见视为否定。
- [ ] 把贡献过程公开记录为 Proof of Work。
- [ ] 持续贡献，而不是把第一个 PR 当作终点。
