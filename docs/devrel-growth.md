## 一句话总结

这场分享用小海自己的路径说明：Web3 职业成长不一定从“纯技术岗位”直接开始，很多人可以先通过智能合约/前端/开源项目/黑客松建立 Builder 证明，再逐步进入社区、运营、生态研究、开发者关系等岗位；DevRel 的核心不是“会说话”，而是能理解开发者、连接生态资源、把项目价值讲清楚，并持续沉淀作品和人脉。

## 嘉宾成长路径

小海的路径大致可以拆成五段：

1. **从智能合约入门**
   - 先学习智能合约，并获得第一份相关工作/实习。
   - 体会到纯智能合约岗位对新人并不友好，通常要求多年经验或很强项目证明。

2. **扩展到前端与全栈能力**
   - 继续学习前端，并在前端方向积累较长时间。
   - 后端有自学经历，但工作经验主要集中在智能合约和前端。

3. **通过 LXDAO 进入社区贡献**
   - 参与 LXDAO 相关贡献。
   - 编写 / 维护过面向新人的以太坊入门教材。
   - 写过区块链文明、以太坊相关长文，并获得较高传播。

4. **从技术转向运营与 DevRel**
   - 后续加入项目做运营与 DevRel，开始从单纯写代码转向生态协作、开发者关系、社区连接。
   - 曾负责以太坊协议奖学金（EPF）华语区相关工作。

5. **进入公链 DevRel**
   - 当前在一条公链担任开发者关系角色，工作内容更接近生态分析、开发者支持、技术叙事、社区和合作方连接。

## DevRel 到底做什么

从分享内容看，DevRel 不是单一岗位，而是技术、内容、运营、BD 和生态研究的交叉。

### 典型工作内容

- 理解项目技术架构、开发者工具和生态定位。
- 研究其他生态或竞品的做法，并反馈给内部开发团队。
- 帮助开发者理解项目、使用工具、完成集成或构建 demo。
- 组织 / 支持活动、分享会、黑客松、线上答疑和社区沟通。
- 写教程、文档、研究文章、案例和项目介绍。
- 连接开发者、项目方、社区、基金会、赞助方和潜在岗位机会。

### DevRel 的能力画像

| 能力 | 为什么重要 |
|------|------------|
| 技术理解 | 至少能读懂协议、SDK、合约、前端或 AI 工具链，才能和开发者有效沟通 |
| 内容表达 | 能把复杂技术和生态机会讲清楚，形成教程、文章、分享和文档 |
| 社区运营 | DevRel 很多时候要组织活动、维护群、回答问题、促进协作 |
| 生态判断 | 需要知道哪些项目、团队、工具、叙事值得连接 |
| 人脉与信任 | Web3 岗位和机会高度依赖长期贡献、关系网络和可验证作品 |
| 英语能力 | 很多基金会、公链、全球项目、会议和面试都需要英文沟通 |

## 给新人选方向的建议

### 1. 纯智能合约岗位对新人偏难

嘉宾明确提到：纯智能合约开发对新人来说机会较少，很多岗位会写 3-5 年甚至 5 年以上经验要求。

更现实的路径是：

- 先学智能合约，建立 Web3 基础理解。
- 同时补前端 / 全栈 / AI coding 能力，做出可展示 demo。
- 通过黑客松、开源贡献、项目实习积累证明。
- 如果短期要找工作，可以考虑从运营、生态、BD、研究助理等更开放的入口进入。

### 2. 运营 / 生态岗位可能是新人入口

运营不是“打杂”，而是 Web3 项目触达用户、开发者和合作方的关键岗位。

常见工作包括：

- 整理信息源、追踪项目动态。
- 发布活动通知、维护社群、组织分享会。
- 做内容传播、X / 公众号 / 小红书等渠道运营。
- 联系项目方、社区、嘉宾和合作伙伴。
- 参与活动 SOP、增长、用户反馈和岗位对接。

对于纯小白，嘉宾建议可以先从运营实习或生态相关工作切入，在工作中继续学习技术和行业知识。

### 3. 前端 / 后端转 Web3 的路径

- 前端同学可以先学习以太坊生态、钱包连接、合约交互、Viem / Ethers、DApp 架构。
- 后端同学可以关注 Go、Python、索引器、数据服务、交易系统、AI agent 与链上交互等方向。
- 但嘉宾也提醒：他本人后端工作经验不多，更多经验来自智能合约、前端和生态岗位。

### 4. Research 方向也有空间

研究向可以关注：

- 协议机制、治理、共识与以太坊路线。
- 项目调研、竞品分析、生态地图。
- 数据分析、DAO 治理、产品研究。
- 将研究转化为文章、报告、公开分享和项目建议。

## 黑客松与项目经验

### 嘉宾的黑客松经验

嘉宾提到自己曾和团队用三天时间做出预测市场类项目，并拿到黑客松冠军。项目大意是围绕 K 线 / 价格预测，通过网格化下注和倍率计算收益。

### 黑客松获奖要点

1. **创意要清楚**
   - 评委需要快速理解你解决什么问题。
   - 不一定要非常复杂，但要有明确场景和叙事。

2. **完整度越高越好**
   - 黑客松不会要求产品完美，但 demo 要能跑。
   - 最好有合约、前端、演示视频、清晰分工和可讲述的用户路径。

3. **UI 很影响第一印象**
   - 嘉宾强调 UI 做得好会明显提高评委第一印象。
   - 如果不会设计，可以用 AI 生成初版，再反复提需求迭代。

4. **项目经验不只看结果，也看你做了什么**
   - 简历里要写清楚：项目是什么、你负责什么、核心逻辑是什么、为什么这么设计。
   - 即便项目没有商业后续，也可以作为经验，但前提是你真正理解它。

### 积累项目经验的方式

- 参加黑客松。
- 给开源项目提 PR。
- 做自己的小工具 / demo / 学习平台。
- 参与实习计划中的小组项目。
- 在生态内找项目方做贡献。

## AI-native Builder 工作流

嘉宾展示了自己的 AI 工作流，并提到现在大量代码由 AI 辅助完成。他的核心观点是：未来写代码不一定要求大量手写，但必须能看懂、审查和控制 AI 生成的代码。

### 对不同 AI 工具的分工理解

| 工具 | 适合场景 |
|------|----------|
| Claude Code | 适合小白从想法生成整体框架，处理较完整的项目骨架 |
| Codex | 适合做更具体、更细节、更明确的编码任务 |
| Gemini | 更适合做 UI / 视觉相关探索 |

### 对新人更重要的能力

- 能把需求拆清楚。
- 能读懂 AI 生成的代码。
- 能 review 逻辑、发现明显错误。
- 能通过提示词和迭代把 demo 做完整。
- 能记录 AI 做了什么、人做了什么判断。

## 英语与全球化机会

嘉宾认为英语很重要，尤其是进入基金会、公链、协议研究、海外项目和全球化 DevRel 时。

几个具体原因：

- 很多项目官网、文档、界面、会议和工作流是英文。
- 面试可能是英文。
- 以太坊、EPF、核心开发者会议等信息源主要是英文。
- 同等条件下，会英文的人可选择的工作机会更多。

学习建议：

- 参加或旁听全英文技术会议，例如以太坊 ACD、执行客户端 / 共识客户端相关会议。
- 看英文字幕的剧集或视频，提高听力输入。
- 在真实主题里练英语，比单独刷听力更容易进步。

## 职业判断与现实提醒

### 市场岗位分布

嘉宾观察到，在当前行情下：

- 市场、BD、运营岗位相对更多。
- 技术岗位整体更少，纯智能合约岗位更少。
- 交易所岗位通常比项目方岗位更多。
- 项目方岗位更难找，但可能更贴近建设者路径。

### 学历与背景

- 学历不是绝对硬门槛，但好学校、好成绩可以成为竞争点。
- 更关键的是项目经验、可验证贡献、表达能力和你在项目中真正做过什么。

### 新人如何养活自己

- 可以先做运营 / 社区 / 生态实习，边工作边学习技术。
- 远程或兼职在 Web3 中相对常见，但也取决于精力和交付能力。
- 最后一周的岗位对接、项目方资源和群内人脉都值得主动利用。

## 对本实习计划学员的行动清单

### 本周立即做

- [ ] 完成 Week 1 钱包、链上交互、基础任务。
- [ ] 开始记录每天学习日志，不只写“做了什么”，也写“卡在哪里、AI 帮了什么、人判断了什么”。
- [ ] 关注至少 5 个 Web3 项目 / 基金会 / 开发者社区的信息源。
- [ ] 选择一个倾向方向：Tech / Ops / Research，但不必过早锁死。

### 两周内沉淀

- [ ] 做一个最小 DApp / 数据看板 / 研究文章 / 活动 SOP。
- [ ] 把成果写成 README 或公开笔记。
- [ ] 尝试给开源项目提一个小 PR，哪怕是文档修正。
- [ ] 整理一版个人介绍：我是谁、我会什么、我做过什么、我想找什么机会。

### Hackathon 前准备

- [ ] 提前找队友：技术、产品、运营、设计 / 路演最好互补。
- [ ] 准备 AI coding 工作流和 UI prompt。
- [ ] 准备 demo 视频模板和 pitch deck 结构。
- [ ] 明确自己在项目中的责任和可讲述贡献。

## 可复用的职业策略

> 先用作品证明自己，再用社区放大机会。

1. **用智能合约建立 Web3 基础，但不要只押注纯合约岗位。**

6. **持续提高英文能力，扩大机会半径。**
